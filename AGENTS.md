# AI Preferences

## Language
- 思考语言：英文（高效模式）
- 输出语言：中文
- 输出中文时使用主动语态，避免长定语从句直译

## Git Commit Messages
- 使用中文撰写 commit message
- 推荐格式：`类型(范围): 描述`
  - `feat`: 新功能
  - `fix`: 修复 bug
  - `refactor`: 重构
  - `docs`: 文档
  - `chore`: 杂项

## Code Modification Principles

### 决策流程
1. **先分析**：问题是什么？影响范围多大？
2. **判断类型**：通用问题还是特例？
3. **评估方案**：会不会增加代码复杂度？会不会破坏现有逻辑？
4. **不确定就问**：不要假设用户意图，有疑问先澄清

### 通用问题 vs 例外处理

**通用问题特征**：多种场景会出现、会影响多个用户

**正确做法**：
- 通用问题用通用方案
- 不因为个别特例添加 if-else 累积到通用逻辑
- 优先从源头解决，而不是事后检测-删除

**反例（不好）**：
```python
# 不好：为了修一个问题加了 workaround，后来又为 workaround 加了补丁
if weird_case_a:
    handle_a()
elif weird_case_b:
    handle_b()
# ...
```

**正例（好）**：
```python
# 好：找到根本原因，用统一的方案解决
def calculate(items):
    return [transform(item) for item in items if item.is_valid()]
```

### 灰色地带处理
当遇到无法归类为纯通用或纯例外的逻辑时：
- 采用"策略模式提取"或"配置化隔离"
- 将特殊逻辑抽离出主流程，而非直接内嵌 if-else

### 代码复杂度信号
当出现以下情况时，方向可能不对，考虑其他方案：
- 需要多 pass 处理（先匹配再删除、先标记再处理）
- 需要保存状态来追踪之前的信息
- 代码行数增加显著（如超过 20%）而功能点没增加

**注意**：此指标主要针对业务逻辑代码，不适用于重构阶段的类型系统建设、抽象基类提取或测试用例覆盖。

### 例外情况处理原则
- 例外情况特殊处理，不累积到通用逻辑
- 如果某个"例外"出现超过 3 次，重新评估是否是通用问题
- 记录例外处理的原因和上下文

### 破坏性修改前必须确认
以下修改必须先跟用户确认：
- 删除现有功能
- 修改核心数据结构
- 改变公共 API
- 引入新的外部依赖

### 版本与冲突
- 本文档版本：`1.1`
- 当本偏好与用户临时需求冲突时，以用户明确指令为准，并记录原因
